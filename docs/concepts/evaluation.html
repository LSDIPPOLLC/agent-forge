Evaluation Pipeline

Every agent iteration runs through a multi-stage evaluation that measures both functional correctness and semantic quality.

Pipeline Stages

Agent Code
    |
 TestGenerator - k test cases (LLM auto-generated)
    |
 TestRunner - exec each test via bun:eval
    |
 Schema validation - pass/fail per test
    |
 LLMJudge - semantic quality score (1-10 per criterion)
    |
 ScoreReport
    |
 Shadow Generalization - 3 unseen test cases
    |
 QualityGate - PASS or FAIL

ScoreReport

interface ScoreReport {
  testPassRate: number;       // 0.0 - 1.0
  judgeScore: number;          // 0.0 - 1.0
  composite: number;           // 0.6 * testPassRate + 0.4 * judgeScore
  generalizationScore: number or undefined; // from shadow eval (0.0 - 1.0)
  generalizationWarning: string or undefined;
  coverageGaps: string[];      // aspects of original intent not tested
  testResults: TestResult[];
  passed: boolean;
}

Test Generation

Given the NL description and agent schemas, the LLM generates k=5 test cases that:

  • Cover all aspects of the original intent
  • Include normal, edge, and error cases
  • Are structurally diverse (Jaccard similarity enforced)
  • Trace to description clauses for coverage tracking

TestRunner

Each test case is executed via bun:eval in an isolated subprocess:

// Agent code is loaded with a runtime shim
 import tools, handle from "agent-forge-runtime"
 agent code here

 // Test input is injected
 const input = { pr_url: "...", repo: "..." };
 const result = await handle(input);

 // Output is validated against expected schema keys
 const hasKeys = expectedKeys.every(k returns k in result);

LLM Judge

For each test output that parses, the LLM judge evaluates semantic quality:

CriterionScore RangeWhat It Measures
Correctness0-25Does the output correctly address the task?
Completeness0-25Are all required fields present and populated?
Quality0-25Is the output insightful and actionable?
Format Compliance0-25Does output match the expected schema?

Judge score = average across criteria / 25

Composite Score

composite = 0.6 * testPassRate + 0.4 * judgeScore

Weight ratio (60/40) favors functional correctness over stylistic quality.

Generalization Holdout

After the main eval passes, 3 additional shadow test cases run against the agent. These are unseen inputs within the intended scope but not similar to any training test case.

If the agent fails the holdout, a warning is attached to the agent record but the agent still passes the gate. This guards against Goodhart's Law: optimizing to the test suite rather than the real use case.
